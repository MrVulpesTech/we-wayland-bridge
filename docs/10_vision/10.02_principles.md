# 10.02 — Principles

Load-bearing principles. Every architectural decision should be checkable
against this list. If a choice violates one, it needs an ADR explaining
why.

---

## P1. Do not rebuild the renderer

`linux-wallpaperengine` already parses, translates, and renders Wallpaper
Engine content. The project's contribution is the output path and the
GNOME integration — not a new wallpaper runtime. When a need looks like
"the renderer should do X", the default is a patch to the renderer
(upstreamable), not a parallel reimplementation.

## P2. Upstream-friendly by default

Renderer changes are structured as if they will become a pull request
into `Almamu/linux-wallpaperengine`: a new output backend that sits
beside the existing ones, not a rewrite of them. Even if upstream
declines the PR, this discipline keeps the fork rebaseable against
upstream `main`. See ADR-0001, ADR-0003.

## P3. Respect the process boundary

The renderer and `gnome-shell` are separate processes and must stay that
way. The renderer is GPLv3, runs untrusted shader code, and crashes are
survivable only if they cannot take the shell down with them. Frames
cross the boundary through an IPC mechanism (PipeWire), never by linking
the renderer into the extension. This boundary is also the licence
boundary (P5).

## P4. Zero-copy or it does not ship

A scene wallpaper at native resolution is a continuous video stream. The
core technical risk is the hop from the renderer's GPU buffer to the
Clutter actor. Software-copying every frame through system memory will
saturate a CPU core at high resolutions and defeat the point. The target
path is GPU-buffer to GPU-buffer via dma-buf, brokered by PipeWire, with
no intermediate copy. This is validated by prototype before it is relied
on (ADR-0004).

## P5. Honour the licence boundary

Code derived from or linked against the GPLv3 renderer is GPLv3. The
GNOME extension is a separate process that exchanges frames over IPC and
links nothing from the renderer; it can therefore carry a permissive
licence (MIT). Keep the boundary clean enough that this separation is
real, not nominal. See ADR-0002.

## P6. The compositor breaks extensions; design for it

GNOME Shell's extension API changes — sometimes incompatibly — at almost
every GNOME release. Assume the extension will break on upgrade. Keep the
GNOME-specific surface small and well-isolated from the rest of the
system, so a port to a new GNOME version is a contained job and the
renderer/transport layers are unaffected.

## P7. Test in a nested shell, never on the live desktop

A misbehaving extension can make the desktop unusable. All extension
testing runs in a nested session
(`dbus-run-session -- gnome-shell --nested --wayland`), not on the
operator's running desktop.

## P8. Read-only on the operator's data

The wallpapers live in the operator's Steam library, reached through
gitignored symlinks. The project reads them. It does not copy them into
the repository, modify them, or redistribute them. The operator must own
Wallpaper Engine and have subscribed to the wallpapers themselves — the
same legal footing on which `linux-wallpaperengine` has operated for
years. See `10.03_non_goals.md`.
