# 10.03 — Non-Goals

What this project will not do, at least through v1. Recording non-goals
is as important as recording goals: it stops scope creep and gives a
clear "no".

## Product scope

- **No web wallpapers (CEF) in v1.** Web-type wallpapers need an embedded
  Chromium (CEF). That is a separate, heavy workstream — deferred, not
  rejected. See `90_open_questions/90.01_open_questions.md`.
- **No `application`-type wallpapers, ever.** These run an arbitrary
  bundled executable. Out of scope on both effort and safety grounds.
- **No audio-reactivity in the MVP.** Scenes that react to system audio
  come after a static scene renders correctly end-to-end.
- **No multi-monitor in the MVP.** One output first; per-monitor
  selection is a later milestone.
- **Not a wallpaper editor or creator.** The project renders existing
  Wallpaper Engine content. Authoring tooling is Wallpaper Engine's job.

## Platform scope

- **GNOME on Wayland is the target.** wlroots compositors already work
  with `linux-wallpaperengine` directly via `wlr-layer-shell`; this
  project does not try to improve or replace that path, only to avoid
  breaking it.
- **KDE Plasma is not this project.** Plasma's sanctioned background path
  is a Plasma wallpaper plugin, a different integration from a GNOME
  Shell extension. If Plasma support happens, it is a separate component,
  not a GNOME subclass.
- **No X11 backend work.** The target system runs Wayland with the X11
  session removed. The existing upstream X11 path is out of scope.

## Distribution and licensing

- **The GNOME Shell extension cannot ship as a Flatpak.** Extension code
  loads inside the `gnome-shell` process; that is architecturally
  incompatible with Flatpak sandboxing. The extension is distributed
  through extensions.gnome.org. The renderer and any GUI manager may be
  Flatpak-packaged separately.
- **No redistribution of Wallpaper Engine content or code.** The project
  ships no wallpaper assets, no `scene.json`, no shader source, and
  nothing from Wallpaper Engine itself. Users bring their own, installed
  through Steam. See `10.02_principles.md` P8.

## Engineering scope

- **No fork of the renderer we maintain in parallel.** Renderer changes
  target upstream (ADR-0001). A long-lived fork is the fallback if the
  PR is declined, not the plan.
- **No reimplementation of scene parsing, shader translation, or `.pkg`
  unpacking.** That is the renderer's job and it already does it.
