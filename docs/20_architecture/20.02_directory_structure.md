# 20.02 — Directory Structure

The repository is a thin shell around an upstream submodule plus the two
pieces of original work (output backend patches, GNOME extension) and the
docs that hold the project together across sessions.

## Top-level layout

```text
we-wayland-bridge/
├── CLAUDE.md            Rules, target system, current state. Read first.
├── README.md           Public-facing overview.
├── upstream/           git submodule: Almamu/linux-wallpaperengine.
│                       Read-only. Never edited in place (writing-style §9).
├── bridge/             Original renderer-side work: the headless→PipeWire
│                       output backend, as patches / a tracked fork branch.
│                       GPLv3 (derived from upstream). See ADR-0002, ADR-0003.
├── extension/          GNOME Shell extension (GJS + Clutter + GStreamer).
│                       Separate process, MIT. See ADR-0002.
├── docs/               Design, architecture, ADRs, session logs.
├── steam-workshop ->   symlink to the user's subscribed wallpapers.
│                       gitignored. Read-only. Never committed.
├── steam-assets ->     symlink to Wallpaper Engine shared assets.
│                       gitignored. Read-only. Never committed.
└── .gitmodules         Pins the upstream submodule.
```

## Why a separate repo and not a fork

The project's product — the bridge plus the extension — is not a
modification of upstream. It consumes upstream as a dependency. Renderer
changes that *are* modifications of upstream live on their own fork branch
and target a pull request. Keeping the top-level repository separate keeps
the licence boundary (ADR-0002) and the upstream relationship (ADR-0001)
clean. Full rationale in ADR-0001.

## The two symlinks

```text
steam-workshop -> ~/.local/share/Steam/steamapps/workshop/content/431960
steam-assets   -> ~/.local/share/Steam/steamapps/common/wallpaper_engine/assets
```

`431960` is the Steam App ID of Wallpaper Engine. The first symlink points
at every subscribed wallpaper (each in its own `<workshop_id>/` directory
with `project.json` and a `scene.pkg`); the second points at the shared
assets the renderer needs to run any scene. Both are operator-private data:
gitignored, read-only, never copied into the repository.
