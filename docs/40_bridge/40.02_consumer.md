# 40.02 — GNOME Extension Consumer

The consumer is a GNOME Shell extension (`extension/`, MIT) that paints the
producer's PipeWire stream into the desktop background. It is the other end
of the pipe from `40.01` (the producer). A separate process from the
GPLv3 renderer; PipeWire IPC only (ADR-0002).

> **Status: VERIFIED end-to-end in a nested session (2026-06-11).** Frames
> flow renderer → PipeWire → extension → `St.ImageContent` → Clutter actor
> in the background layer (`wwb: first frame uploaded`, samples streaming).
> Proven on GNOME Shell 50.1 via the Mutter Devkit. Tested headless
> (`--virtual-monitor`), so not yet *watched* on a visible display — that
> and production polish remain (see §5).

## 1. Background-layer injection (from the study)

Studied `gnome-ext-hanabi` and `@kv9898`'s fork (the two working GNOME
live-wallpaper approaches). Both place content in the background by
overriding `Background.BackgroundManager._createBackgroundActor` and
`add_child`-ing their actor to each monitor's background actor. We reuse
that — it is the only sanctioned background-layer path on GNOME Wayland.

**Key simplification from the PipeWire approach:** Hanabi and kv9898 render
into a real (hidden) toplevel window and `Clutter.Clone` it, so they need
~10 overrides to hide that window from the overview, Alt-Tab, the dock, the
app list, and the window tracker. **Our producer is a separate process with
no shell window**, so the consumer needs *none* of that — just the one
injection point. This is the structural payoff of going through PipeWire
instead of window-cloning, and a concrete differentiator for Q-10.

## 2. Frame path

```text
producer ──PipeWire(BGRx)──► pipewiresrc ! videoconvert !
  video/x-raw,format=RGBA ! appsink   (inside gnome-shell)
     │  GLib timer @30Hz on the MAIN thread: try_pull_sample
     ▼
  St.ImageContent.set_bytes  ──►  LiveWallpaper (St.Widget) in the
                                   monitor's background actor
```

- **`appsink` needs `import GstApp`.** Without it the appsink is a plain
  `Gst.Element` and `try_pull_sample` is unbound — it throws on every poll.
  This cost a debugging round (the throw was being swallowed). Import
  `gi://GstApp` for the side effect of registering `GstAppSink` methods.
- **Threading:** GStreamer's streaming thread must not touch Clutter/GJS.
  The sink is polled from a GLib timer on the shell main thread
  (`try_pull_sample(0)`), so every Clutter call is main-thread. No
  cross-thread GJS.
- **Upload (verified API on GNOME 50.1):**
  `St.ImageContent.set_bytes(coglContext, bytes, Cogl.PixelFormat.RGBA_8888,
  w, h, stride)` — note the **leading `CoglContext`** (6 args; the docs/older
  examples show 5). The CoglContext is
  `global.stage.context.get_backend().get_cogl_context()` (the extension
  tries several paths and logs the one that works).
- **PipeWire target:** a bare `pipewiresrc` is **not** auto-linked to a
  `Video/Source` by WirePlumber (camera-like policy), so it must name the
  producer's node (`path=<id>`). Currently the node id comes from the
  `WWB_PIPEWIRE_PATH` env var for testing; production needs node-by-name
  discovery (see §5).
- **Fallback colour** (`#0a0f1e`) until the first frame, so injection is
  verifiable independently of the upload path.
- **Reconnect:** a bus watch on ERROR/EOS rebuilds the pipeline after 2 s
  (producer restart).

## 3. Requirements carried from 40.01

The producer docs set consumer requirements (per-monitor size; fill/fit/
crop; opaque BGRx). Current state:

- **Per-monitor size:** each `LiveWallpaper` reads its monitor size and
  sizes the actor; implemented, untested.
- **Fit modes:** MVP stretches (`Clutter.ContentGravity.RESIZE_FILL`).
  `RESIZE_ASPECT` gives letterbox-fit; cover/crop is a follow-up.
- **Opaque:** the stream is BGRx → RGBA via `videoconvert`; treated opaque.
- **Run the producer at 1080p+** or scenes wash out (Q-12).

## 4. Running a nested shell (GNOME 50.1)

`--nested` was removed in GNOME 49; the nested shell is now the **Mutter
Development Kit**. The working invocation (operator runs it from a **real
terminal**, not the snap-confined VS Code one):

```sh
dbus-run-session -- gnome-shell --wayland --devkit --virtual-monitor 1920x1080
```

Gotchas learned:
- `--nested` → "Unknown option"; bare `--wayland` → `EBUSY` (tries to seize
  the live seat); `--headless` → gsettings schema abort in the sandbox.
- The Devkit starts with **0 monitors** unless given `--virtual-monitor`;
  with 0 monitors there are no background actors and nothing to inject into.
- `--virtual-monitor` is **headless** (no on-screen window), so the pipeline
  can be verified by logs but not watched. A visible nested display is the
  open item in §5.
- The assistant must **never** launch or `pkill` gnome-shell — it shares the
  operator's live session (a `pkill gnome-shell` once logged the operator
  out).

## 5. Done vs. remaining

**Done (verified):** background-layer injection; per-monitor `LiveWallpaper`;
GStreamer pull on the main thread; `St.ImageContent` upload; continuous
frames from the producer.

**Remaining:**
1. **Watch it.** `--virtual-monitor` is headless. Either find a visible
   Devkit display, or test on the live session (the extension is contained:
   errors are caught and it reverts on disable; keep
   `gnome-extensions disable …` ready as the escape hatch).
2. **Producer discovery.** Replace the `WWB_PIPEWIRE_PATH` env var with
   discovery of the producer by `node.name` (`wpe-host`), so the consumer
   finds it without a hardcoded id.
3. **Auto-enable was flaky** across nested runs; explicit
   `gnome-extensions enable` was reliable. Re-check on a stable session.
4. Fit/crop modes (`RESIZE_ASPECT` etc.), multi-monitor, fullscreen-pause
   (battery), and producer lifecycle (autostart/stop with the extension).
