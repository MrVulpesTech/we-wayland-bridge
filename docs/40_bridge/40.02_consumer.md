# 40.02 — GNOME Extension Consumer

The consumer is a GNOME Shell extension (`extension/`, MIT) that paints the
producer's PipeWire stream into the desktop background. It is the other end
of the pipe from `40.01` (the producer). A separate process from the
GPLv3 renderer; PipeWire IPC only (ADR-0002).

> **Status: VISIBLE AND ANIMATING on the live desktop (2026-06-14).** A scene
> renders as the real GNOME background on Shell 50.1. The architecture and the
> rendering path below are the current, working ones — they replaced the
> first-cut `St.ImageContent` design after the freeze incident (`40.04`) and the
> live rendering bugs (`session-04-live-wallpaper.md`). **Known limit:** laggy at
> native 2560×1440 (the SHM-copy upload cost — Stage C/dma-buf is the fix). See
> §5.

## 1. Background-layer injection (from the study)

Studied `gnome-ext-hanabi` and `@kv9898`'s fork (the two working GNOME
live-wallpaper approaches). Both place content in the background by
overriding `Background.BackgroundManager._createBackgroundActor` and
`add_child`-ing their actor to each monitor's background actor. We reuse
that — it is the only sanctioned background-layer path on GNOME Wayland.

**Key simplification from the PipeWire approach:** Hanabi and kv9898 render
into a real (hidden) toplevel window and `Clutter.Clone` it, so they need
~10 overrides to hide that window from the overview, Alt-Tab, the dock, the
app list, and the window tracker. **Our producer is a separate process with
no shell window**, so the consumer needs *none* of that — just the one
injection point. This is the structural payoff of going through PipeWire
instead of window-cloning, and a concrete differentiator for Q-10.

## 2. Frame path

One **shared `FrameSource`** owns the only pipeline, poll timer, and texture for
the whole extension. Each background actor is a thin, pipeline-less
`LiveWallpaper` that paints the shared texture.

```text
producer ──PipeWire(BGRx)──► pipewiresrc ! videoconvert !
  video/x-raw,format=RGBA ! appsink        (one FrameSource, in gnome-shell)
     │  GLib timer @~15Hz on the MAIN thread: try_pull_sample
     ▼
  Cogl.Texture2D (allocated once, set_data per frame, RGBX)
     │  queue_redraw() on each subscriber
     ▼
  LiveWallpaper.vfunc_paint_node → Clutter.TextureNode   (background actor)
```

- **One source, many actors.** `_createBackgroundActor` fires on every overview
  toggle, workspace switch and monitor change. A pipeline *per actor* leaked a
  pipeline + timer each time and crashed the shell (`40.04` Failure 4). The
  source is created once, started deferred, stopped on disable; actors
  `subscribe()`/`unsubscribe()` and are cheap to churn.
- **Cleanup via the `destroy` signal, not a `destroy()` override.** The shell
  disposes actors as children without calling a JS `destroy()` override, so
  cleanup keyed on the override leaked subscriptions and produced "already
  disposed" criticals. Connect the `destroy` signal instead.
- **`appsink` needs `import GstApp`** or `try_pull_sample` is unbound and throws
  every poll (swallowed silently → samples=0).
- **Threading:** the sink is polled from a GLib timer on the shell main thread
  (`try_pull_sample(0)`); no GStreamer-thread Clutter access.
- **Texture upload (reuse, no per-frame allocation):** allocate one
  `Cogl.Texture2D.new_from_data(ctx, w, h, fmt, stride, bytes)` on the first
  frame, then `texture.set_data(fmt, stride, bytes, 0)` in place every frame.
  The CoglContext is `global.stage.context.get_backend().get_cogl_context()`.
- **Opacity: upload as `Cogl.PixelFormat.RGBX_8888`, not RGBA.** The producer is
  BGRx and `videoconvert` to RGBA leaves **alpha = 0**, so an RGBA texture
  paints fully transparent (the "nothing on screen" live bug). RGBX ignores the
  alpha byte ⇒ always opaque; RGB bytes sit in the same positions.
- **Painting (live-verified facts):** a `LiveWallpaper` has no CSS background and
  no Clutter content, so it must (a) override `vfunc_get_paint_volume` to claim
  its allocation — otherwise Clutter culls it and `vfunc_paint_node` stops being
  called; and (b) build `Clutter.TextureNode(texture, white, LINEAR, LINEAR)`
  with an **explicit opaque white `Cogl.Color`** (a null colour paints
  transparent), `add_rectangle` over the allocation, `node.add_child`.
- **Throttle + overview pause:** poll at ~15 fps (`FRAME_INTERVAL_MS`); pause all
  pulls while the overview is open (`Main.overview` `showing`/`hidden`) — the
  overview's background blur stacked on the upload hard-locked the iGPU
  (`40.04` Failure 2).
- **PipeWire target:** a bare `pipewiresrc` is **not** auto-linked to a
  `Video/Source` (WirePlumber camera policy). The source finds the producer by
  `node.name=wpe-host` via `Gst.DeviceMonitor` and builds from that
  `Gst.Device`. `WWB_PIPEWIRE_PATH=<id>` is a debug override.
- **Reconnect / teardown:** a bus watch on ERROR/EOS rebuilds after 2 s;
  teardown drives the pipeline to NULL and **waits** for it before releasing the
  ref (disposing a PLAYING pipewiresrc SIGSEGV'd the shell — `40.04` Failure 3).
- **Kill-switch:** `~/.config/wwb/disabled` makes `enable()` a no-op
  (`WWB_FORCE=1` bypasses it for nested testing).

## 3. Requirements carried from 40.01

The producer docs set consumer requirements (per-monitor size; fill/fit/
crop; opaque BGRx). Current state:

- **Per-monitor size:** each `LiveWallpaper` reads its monitor size and
  sizes the actor; implemented, untested.
- **Fit modes:** MVP stretches the texture over the actor allocation (the
  `add_rectangle` box). Aspect/cover/crop is a follow-up — and matters because
  lowering the producer below native res changes the scene framing (the renderer
  frames the scene to its output size), so res is not a free knob.
- **Opaque:** see §2 — upload as RGBX; RGBA would be transparent (alpha 0).
- **Run the producer at 1080p+** or scenes wash out (Q-12).

## 4. Running a nested shell (GNOME 50.1)

`--nested` was removed in GNOME 49; the nested shell is now the **Mutter
Development Kit**. The working invocation (operator runs it from a **real
terminal**, not the snap-confined VS Code one):

```sh
dbus-run-session -- gnome-shell --wayland --devkit --virtual-monitor 1920x1080
```

Gotchas learned:
- `--nested` → "Unknown option"; bare `--wayland` → `EBUSY` (tries to seize
  the live seat); `--headless` → gsettings schema abort in the sandbox.
- The Devkit starts with **0 monitors** unless given `--virtual-monitor`;
  with 0 monitors there are no background actors and nothing to inject into.
- `--virtual-monitor` is **headless** (no on-screen window), so the pipeline
  can be verified by logs but not watched. A visible nested display is the
  open item in §5.
- The assistant must **never** launch or `pkill` gnome-shell — it shares the
  operator's live session (a `pkill gnome-shell` once logged the operator
  out).

## 5. Done vs. remaining

**Done (live-verified, 2026-06-14):** background-layer injection; shared
`FrameSource`; producer discovery by `node.name`; reused-texture upload (RGBX);
`TextureNode` painting with the cull/colour fixes; overview pause; clean disable
(no GStreamer criticals). A scene animates on the real desktop.

**Remaining, in priority order:**
1. **Stage C — dma-buf zero-copy.** The blocker for real use: at native
   2560×1440 the per-frame ~14 MB SHM upload on the main thread makes the
   desktop laggy. Stage C removes the copy (GPU samples the producer buffer) and
   also retires libgstpipewire-in-shell. This is the priority.
2. **Fit/crop modes** — stretch-only today; needed before lowering producer res
   is usable (framing changes with render size).
3. Multi-monitor; fullscreen-pause (battery / let games have the GPU); producer
   lifecycle (autostart/stop with the extension).
4. Auto-enable-at-login was flaky in early nested runs; explicit
   `gnome-extensions enable` is reliable. Re-check.
