# 40.05 — Stage C: dma-buf zero-copy

Stage C removes the per-frame copy that makes the SHM path lag at native
resolution (40.04): the producer renders into a dma-buf and the consumer
samples it on the GPU, with no `glReadPixels` on the producer and no
`set_data` upload on the shell main thread.

Built in checkpoints. **C-1 and the C-2 producer are done and verified; the
C-2/C-3 consumer is blocked on a real platform limit** (below).

## C-1 — producer renders into a dma-buf (done, verified 2026-06-14)

`--dmabuf` backs the output FBO with a gbm-allocated buffer imported as an EGL
image (`bridge/host/main.cpp`):

- gbm device on `/dev/dri/renderD128`; `gbm_bo_create_with_modifiers` with
  `DRM_FORMAT_MOD_LINEAR` (accepted first try on CometLake — no fallback).
- Imported via `eglCreateImageKHR(EGL_LINUX_DMA_BUF_EXT)` →
  `glEGLImageTargetTexture2DOES` → bound as the FBO colour attachment.
- Exported and cross-checked via `eglExportDMABUFImageMESA`.

Negotiated: `XBGR8888` (`XB24`), `LINEAR`, stride 7680 = 1920×4 (tightly
packed). Frame 5 PNG correct (astronaut, right colours/orientation). Render cost
unchanged vs Stage A (~30 ms/frame); the win is removing the readback.

**Format:** `XBGR8888`, not `ABGR8888`. The scene renders alpha=0; an alpha
format makes GL consumers premultiply the RGB to black. The X (ignore-alpha)
format is opaque and matches the `RGBx` wire format.

## C-2 — dma-buf PipeWire transport (producer done; gst consumer sync caveat)

`--dmabuf` publishes the frames as dma-buf PipeWire buffers; `--shm` forces the
Stage B `glReadPixels`→SHM path (verified still working).

- A **pool of 4 dma-bufs** (each its own gbm_bo + EGL image + FBO), rendered
  round-robin (`wp_project_set_output_framebuffer` per frame). A single shared
  dma-buf deadlocks: the consumer holds the only buffer, the producer's
  `dequeue` returns null and never renders the buffer being read → the consumer
  sees an empty buffer.
- `SPA_DATA_DmaBuf`, `RGBx` + mandatory `LINEAR` modifier in the `spa_pod`,
  `PW_STREAM_FLAG_ALLOC_BUFFERS`, fd dup'd per buffer in `add_buffer`.

**Verified:** gst negotiates a **direct** dma-buf import
(`gst_egl_image_from_dmabuf_direct_target_with_dma_drm`: `fourcc=XB24`,
`pitch=7680`, `modifier=0`, `DirectDmabuf returned 1`); the pool streams; and the
producer's dma-buf **memory is provably correct** — `gbm_bo_map` (CPU) reads the
rendered pixels (`235,238,243`) at stride 7680, matching the GL view.

**Caveat — the `gst-launch` demo renders black.** Not a producer bug: the buffer
memory is correct and gst imports it correctly. The consumer's GPU *samples*
black because the producer's GPU writes are not fenced against the consumer's
GPU reads. `glFinish` flushes the producer queue and makes the buffer
CPU-coherent (so `gbm_bo_map` sees it) but does not attach an implicit fence the
consumer waits on. A one-shot `glupload` on a separate GL context with no fence
is the worst case for this. The actual target (a compositor's continuous render
loop with i915 implicit sync) is different — but see C-3.

## C-3 — consumer-side import: BLOCKED (no GJS path)

The plan was to import the dma-buf fd into Cogl in the extension
(`eglCreateImageKHR` → wrap as a Cogl texture) and paint it as before. **This is
not reachable from a GJS extension on GNOME 50.1 / Mutter-18:**

- **GJS has no EGL bindings** — `eglCreateImageKHR` cannot be called.
- **Cogl/Mutter expose no dma-buf import to introspection.** Checked the
  Mutter-18 typelibs (`Cogl-18`, `Meta-18`, `St-18`) by string dump and via
  `gjs`:
  - `cogl_egl_texture_2d_new_from_image` takes an `EGLImage` (non-introspectable)
    → excluded from the typelib.
  - `CoglDmaBufHandle` is **export-only** (allocate + `get_fd`/`get_framebuffer`;
    used by Mutter for screencast) — no `new_from_fd`.
  - `MetaMultiTexture` takes already-imported Cogl planes, not a dma-buf fd.
  - `St.ImageContent` only does CPU `set_bytes`/`set_data`.

So the consumer cannot turn a dma-buf fd into a paintable texture in pure GJS.

### Options (operator decision)

1. **Native introspectable helper** (C, MIT): a small `.so` + typelib the
   extension loads, exposing `texture_from_dmabuf(cogl_ctx, fd, w, h, fourcc,
   modifier, stride, offset) → Cogl.Texture`. Real zero-copy, but it must link
   **`libmutter-cogl-18`** to call `cogl_egl_texture_2d_new_from_image` with the
   shell's CoglContext — a **private, version-pinned ABI** (rebuild per GNOME
   release). Adds a build step and breaks the drop-in pure-GJS extension model.
2. **Window-clone architecture** (Hanabi / @kv9898 style): if the renderer drew
   into a real hidden window, Mutter would already hold its texture and a
   `Clutter.Clone` is zero-copy **for free** — no import needed. Cost: the ~10
   overview/Alt-Tab/window-hiding overrides we avoided, and a renderer window.
   This is the Q-10 trade-off made concrete: window-clone gets zero-copy by
   piggybacking on Mutter's own dma-buf import; our headless-PipeWire design has
   to re-import, which GJS can't do.
3. **Stay on SHM** (current, working): keep the elevated VmRSS / native-res lag;
   ship the dma-buf producer behind `--dmabuf` for a future consumer.

The producer dma-buf work (C-1 + C-2 producer) stands regardless of the choice —
it is the half that was in doubt (Q-1: can we render zero-copy and export a
dma-buf), and it works. The remaining question is purely how the GNOME consumer
imports it.
