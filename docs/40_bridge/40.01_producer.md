# 40.01 — PipeWire Frame Producer

The producer is a host program that embeds `linux-wallpaperengine-core`
(from the pinned `next-v2` commit), renders a wallpaper offscreen, and —
in later stages — publishes the frames as a PipeWire video stream. It
lives in `bridge/` and is GPLv3 (it links the GPLv3 core; ADR-0002).

Built in stages with hard checkpoints. **Stage A is complete** (offscreen
render via the core embedding API). Stages B (PipeWire/SHM) and C
(dma-buf zero-copy) follow.

## Prerequisites

Renderer build deps from `50_build_and_run/50.01_system_requirements.md`,
plus what the host links: `libegl-dev`, `libgl-dev`, `libglew-dev`
(all already installed on the reference machine). `libgbm-dev` is needed
for Stage C.

## One-command build

From the repository root, with the worktree already present:

```sh
bridge/build.sh
```

This (idempotently) initializes the worktree's submodules, builds core
with `webhelper`/`frontend`/`dev-viewer`/`tests` disabled, then builds the
host. Artifacts go to `bridge/build/` (gitignored). To wipe and redo:
`rm -rf bridge/build && bridge/build.sh`.

### Recreating the worktree from scratch

If `next-v2-review/` is gone:

```sh
cd upstream
git fetch origin pull/609/head:next-v2-review
git worktree add ../next-v2-review next-v2-review
cd ..
bridge/build.sh
```

### Why core builds without the rest

`next-v2` exposes CMake options (`WPBUILD_WEBHELPER`, `WPBUILD_FRONTEND`,
`WPBUILD_DEVVIEWER`, `WPBUILD_TESTS`); each sibling `CMakeLists.txt` does
`if(NOT WPBUILD_X) return()`, so disabling them skips their configure —
including the webhelper's CEF download. Core talks to the web browser over
IPC, so it does not link CEF; scene and video wallpapers build, web does
not (out of MVP scope anyway).

## Run

```sh
bridge/build/host/wpe-host \
  --assets-dir steam-assets \
  --out bridge/out \
  --width 1920 --height 1080 --frames 120 --fps 60 \
  steam-workshop/2804205787
```

Flags: `--assets-dir` (Steam shared assets), `--out` (PNG dir),
`--width`/`--height` (output FBO size), `--frames`, `--fps` (the virtual
clock rate), `--readback-all` (benchmark: time per-frame `glReadPixels`),
positional wallpaper folder. Diagnostic env var `WPE_FIXED_TIME=<seconds>`
freezes the clock (used in verification below).

## How it works (Stage A)

1. **Headless EGL.** `EGL_PLATFORM_SURFACELESS_MESA` display, OpenGL 3.3
   core context, `eglMakeCurrent` with `EGL_NO_SURFACE`
   (`EGL_KHR_surfaceless_context`). No window, no compositor surface.
   Chosen over gbm because it needs no DRM render-node handling and works
   on the reference Intel/Mesa stack; gbm becomes necessary in Stage C
   where the output texture must be dma-buf backed.
2. **Host-owned FBO.** The host creates an RGBA8 texture + FBO at the
   requested size and hands the FBO id to core via
   `wp_project_set_output_framebuffer`. Core renders into it; the host
   owns the backing texture (the seam that makes Stage C zero-copy
   possible).
3. **Embedding API** (`30_rendering/30.02_next-v2-core-api.md`):
   `wp_config_create` → `wp_context_create` →
   `wp_context_set_gl_proc_address` (host's `eglGetProcAddress`) →
   `wp_context_set_time_counter` (host's virtual clock) →
   `wp_project_load_folder` → `wp_project_set_output_framebuffer` → per
   frame `wp_render_update_time` + `wp_render_frame`.
4. **Virtual clock.** The time callback returns `frame / fps`, so a fixed
   frame count shows real animation regardless of wall-clock render speed,
   and runs are reproducible.

## Verification (Stage A)

Command (astronaut scene, 120 frames at 1920x1080):

```sh
bridge/build/host/wpe-host --assets-dir steam-assets --out bridge/out \
  --width 1920 --height 1080 --frames 120 --fps 60 steam-workshop/2804205787
```

Outputs `bridge/out/frame_{001,060,120}.png`. A second colorful sample
(Frieren scene) is in `bridge/out/frieren/`.

**Animation is real and clock-driven — proven, not eyeballed:**

| Test | Result | Meaning |
|---|---|---|
| `time_cb_calls` | 120 (one per frame) | core uses the host clock |
| Frozen clock (`WPE_FIXED_TIME=1.0`), frame 1 vs 120 | mean abs diff **0.000** | render is fully deterministic per virtual time |
| Normal clock, frame 1 vs 120 | mean abs diff **> 0** | the differences come from advancing time, i.e. animation |

The astronaut scene animates subtly (drifting flare smoke over snow), so
the per-pixel difference is small but non-zero; the frozen-vs-normal
contrast is the proof. The Frieren sample is the easy visual check.

### Measured render performance

Reference machine (Intel UHD CometLake, Mesa), 1920x1080 output, `glFinish`
per frame so these are real GPU times. The `--readback-all` flag adds a
per-frame `glReadPixels` (the GPU→CPU copy Stage B pays and Stage C
removes) and times it separately:

| Scene | Intrinsic size | Render only | + readback | Readback cost |
|---|---|---|---|---|
| Astronaut (`2804205787`) | 2560x1440 | ~27 ms/frame (36 fps) | ~34 ms/frame (29 fps) | **~6.7 ms/frame** |
| Frieren (`3622495963`) | 5160x2160 | ~105 ms/frame (9.5 fps) | ~110 ms/frame (9.1 fps) | **~5.0 ms/frame** |

A 1920x1080 RGBA frame is ~8 MB; reading it back costs ~5–7 ms here.
Against a 60 fps budget (16.7 ms/frame) that is ~30–40 % on a light scene
— **this is the motivation for Stage C** (dma-buf), which removes the
readback on the producer side and the matching upload on the consumer
side. Stage B will pay this cost knowingly for correctness first.

These are first measurements on one iGPU; not a performance claim
(writing-style §6). The Frieren scene is a 5160x2160 puppet-mesh +
lighting scene — a worst-case sample, not typical.

## Known behaviour and issues

- **Scene intrinsic size vs output size.** `wp_project_get_width/height`
  return `1` until the first `wp_render_frame` (core creates the
  renderable lazily). After that they report the scene's authored size
  (astronaut 2560x1440, Frieren 5160x2160). The host renders into its own
  FBO size and the scene scales to it. `wp_project_hint_size` only affects
  web wallpapers in core, so for scenes the **output FBO size is what sets
  resolution.**
- **Aspect ratio.** Forcing 1920x1080 (16:9) on the Frieren scene
  (5160x2160 ≈ 2.39:1) makes it look cropped/zoomed. Matching the FBO to
  the scene or target-output aspect avoids this; deferred until the
  consumer (Session 4) drives the size.
- **Alpha is 0 — frames are opaque RGB but transparent.** Core renders
  the scene's colour but leaves the alpha channel at the cleared 0, so a
  raw RGBA capture is fully transparent and shows blank in normal image
  viewers (the RGB is intact underneath). A wallpaper is opaque, so the
  host forces alpha to 255 when writing PNGs. **Stage B's PipeWire stream
  must use an alpha-less format (BGRx/RGBx)** or force alpha for the same
  reason — a consumer fed BGRA would see a transparent stream.
- **No `glClear`.** The scene fills the frame, so the FBO is not cleared.
- **Diagnostics retained.** The time-call counter and `WPE_FIXED_TIME`
  freeze are kept as cheap debug aids.

## Requirements this places on the Session 4 consumer

The intrinsic-size / aspect findings above are not just producer quirks —
they define behaviour the GNOME extension (Session 4) must own:

1. **Per-monitor output size.** Scenes have an authored size unrelated to
   the display (astronaut 2560x1440, Frieren 5160x2160). The output
   resolution is set by the producer's FBO size, which must be **driven by
   the consumer**: one producer instance (or stream) per monitor, sized to
   that monitor's resolution and scale. The producer must accept the
   target size from the consumer rather than hardcode it.
2. **Fill / fit / crop modes.** Because scene aspect rarely matches the
   monitor (Frieren is ~2.39:1 onto a 16:9 panel), the consumer must offer
   the same scaling modes Wallpaper Engine does — **fill** (cover, crop
   overflow), **fit** (letterbox), **stretch** — and a default. Whether
   the scaling happens producer-side (FBO/viewport) or consumer-side
   (Clutter actor) is a Session 4 design decision; the contract must carry
   enough information (scene aspect, chosen mode) either way.
3. **Opaque frames.** The stream is opaque (BGRx); the consumer must not
   treat it as having a usable alpha channel (see the alpha note above).

Captured as a requirement, not a "nice to have": getting this wrong makes
every non-16:9 scene look cropped or stretched on every monitor.

## Status and next stages

- **Stage A: done.** Offscreen render via core embedding, verified.
- **Stage B (next):** add a PipeWire output. `glReadPixels` into SHM
  buffers is acceptable here (copy path); target a format a GStreamer sink
  consumes; add proper stream metadata and clean shutdown. Verify with
  `gst-launch-1.0 pipewiresrc ! videoconvert ! autovideosink` showing the
  live wallpaper, and log host CPU at 1080p60.
- **Stage C (stretch):** back the output texture with a gbm/dma-buf
  allocation, export via `eglExportDMABUFImageMESA`, negotiate dma-buf
  buffers on the PipeWire stream; keep SHM as a `--shm` fallback. This is
  where Q-1 (zero-copy) is actually proven.
