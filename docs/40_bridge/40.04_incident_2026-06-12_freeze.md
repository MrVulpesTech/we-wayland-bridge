# 40.04 — Incident: hard freeze + grey-screen login (2026-06-12)

During the first live-session test (40.03), the machine **hard-froze** (power-
button reset required). On the next boot, after the password the shell never
appeared — **grey screen with a movable cursor**. Recovered by moving the
extension out of `~/.local/share/gnome-shell/extensions/` and rebooting.

This is a serious regression. The extension must never be able to do this
again. Two distinct failures; one is proven by the logs, one is a hypothesis.

## Evidence

Two journals were captured from the **post-reset** boots (not the original
freeze, which could not log — see below):

| File | Boot | What it shows |
|---|---|---|
| `freeze-boot.log` | 01:41 | The **grey-screen** boot: extension auto-enabled, shell main thread blocked. |
| `grey-boot.log` | 01:50 | The **recovery** boot: extension removed, shell starts clean (no `wwb` lines). |

(The filenames are effectively swapped vs. their contents; the data is what
matters.)

## Failure 1 — grey screen (PROVEN, the dangerous one)

In `freeze-boot.log`, the user shell (`gnome-shell[7253]`) starts at 01:42:27
and logs, at 01:42:29:

```text
wwb: API probe — St.ImageContent=yes …
wwb: _createBackgroundActor fired (monitor 0)
wwb: LiveWallpaper on monitor 0 (2560x1440), bgActor 2560x1440 visible=true opacity=255
```

…and then **nothing more from pid 7253 — zero further lines**. The very next
thing `LiveWallpaper._init` does after that log line is call `this._start()`.
So the main thread blocked inside `_start()`.

The cascade confirms a hung shell:

```text
01:44:29  Failed to activate service 'org.gnome.Shell.Screencast'
01:46:00  xdg-desktop-portal.service: start operation timed out. Terminating.
01:46:00  xdg-desktop-portal-gnome.service: start operation timed out. Terminating.
01:46:29  Failed to activate 'org.freedesktop.portal.Desktop' / '…impl.portal.desktop.gnome'
01:47:15  xdg-desktop-portal-gtk.service: start operation timed out. Terminating.
```

Mutter keeps compositing (the cursor moves), but `gnome-shell`'s GJS main loop
is blocked, so the UI never draws and every service that waits on
`org.gnome.Shell` times out. That is the grey screen.

**Root cause.** `_start()` runs, synchronously on the shell main thread:
`ensureGst()` → `Gst.init()` (first GStreamer use in the process; scans the
plugin registry), then `findProducerDevice()` →
`new Gst.DeviceMonitor()` → `monitor.start()` → `get_devices()`. With **no
producer running** (this was a fresh login — the producer must be started
manually), this work blocked the main loop. Because the extension was in
`enabled-extensions`, **every login re-ran it**, so the grey screen recurred
on each boot — which is why recovery required physically removing the
extension directory, not just `gnome-extensions disable` (you cannot reach a
terminal in the GNOME session when the shell is grey).

**Why the headless nested tests never caught it:** in every nested test the
producer was already running, so the device scan returned immediately. The
no-producer path — the normal login path — was never exercised.

## Failure 2 — the original hard freeze (HYPOTHESIS, not in logs)

The hard freeze happened during the live test while video was actually flowing
(producer streaming ~30 fps, extension uploading frames). A true freeze cannot
write its own journal, and neither captured log covers that moment; there is
**no i915 GPU-hang, no OOM-kill, no soft-lockup** recorded. So this is a
hypothesis, ranked:

1. **Per-frame GPU/texture churn (most likely).** `_uploadFrame` runs at
   ~30 Hz and does `new GLib.Bytes(buffer)` + `St.ImageContent.set_bytes(...)`
   **every frame** at the monitor's native size. At 2560×1440 RGBA that is
   ~14 MB/frame ≈ **420 MB/s** of fresh buffers/textures. If `set_bytes`
   allocates a new Cogl texture each call and GJS GC lags behind, system and
   GPU memory balloon within seconds → exhaustion → on integrated i915 this
   can wedge the GPU and hard-lock the display. Fits "froze shortly after the
   video appeared."
2. **Combined GL load on the iGPU.** The compositor, the producer's EGL
   context, and GStreamer `videoconvert` + texture upload all hammer one
   CometLake UHD GPU; an i915 driver hang under that load is plausible and
   would not necessarily log before the lock.
3. Less likely: the same main-loop block compounding under load.

This must be treated as real until disproven. The hardening caps the churn;
Stage C (dma-buf zero-copy) would remove the per-frame copy/upload entirely
and is the proper long-term fix.

## What was NOT the cause

- No `oom-kill` / `Out of memory` in either log (the "oom" hits are the
  `systemd-oomd` service name only).
- No kernel `GPU HANG` / `Resetting` / `wedged` / soft-lockup logged.
- Not a producer crash (the producer is a separate process; it cannot freeze
  the shell directly).

## Minimal repro — nested only, NO live session

Reproduce **Failure 1** safely in a Mutter Devkit nested session (40.03 §4),
which is disposable:

1. Ensure **no producer is running**.
2. Start the nested shell with the extension enabled:
   `dbus-run-session -- gnome-shell --wayland --devkit --virtual-monitor 1920x1080`
3. Before hardening: the nested shell hangs grey, `wwb` log stops at
   `LiveWallpaper …` (same signature). After hardening: the shell comes up
   normally, logs `wwb: no producer yet, idling` (or similar), and stays
   responsive; starting the producer later makes the wallpaper appear.

Do **not** attempt to reproduce Failure 2 on the live session. If needed,
exercise the upload path in the nested session and watch memory
(`while true; do grep VmRSS /proc/$(pgrep -f devkit)/status; sleep 1; done`).

## Failure 3 — gnome-shell SIGSEGV in libgstpipewire (NEW, the likely real cause)

Verifying the hardening in a nested session surfaced the probable real cause
of the original incident. With the grey-screen fix in place, the nested shell
came up responsive with no producer, idled, then streamed frames correctly —
**but on teardown (producer `Ctrl-C`'d + shell `Ctrl-C`'d) gnome-shell
crashed**:

```text
Signal: 11 (SIGSEGV)   SegvReason: reading NULL VMA
StacktraceAddressSignature: …/gstreamer-1.0/libgstpipewire.so+15fa1
(just before) Trying to dispose element wwb-pipeline, but it is in PLAYING
              instead of the NULL state
```

So the GStreamer **PipeWire plugin running inside gnome-shell** segfaults when
the pipeline is finalized while still PLAYING — i.e. on producer disappearance
or shell shutdown. On Wayland a shell crash **ends the session**, which is what
the original "hard freeze" most likely was.

Immediate fix: tear the pipeline down cleanly — remove the bus handler, set
`NULL`, and **wait** for the `NULL` transition (`get_state`) before releasing
the reference, so libgstpipewire is fully stopped before GC. This must be
re-verified for two operational cases, both in a nested session:

1. **Clean disable while streaming** (`gnome-extensions disable` with the
   producer running) — the shell must survive.
2. **Producer stops while streaming** (`Ctrl-C` the producer) — the extension
   must reconnect without crashing the shell.

(The first crash was on `Ctrl-C` of the nested shell, which is an abrupt kill;
the two cases above are what actually matter for the live session.)

**Re-verified 2026-06-12:** with the clean-teardown fix, both cases pass in a
nested session — producer-disconnect and clean disable both leave the shell
ALIVE, no SIGSEGV, no `dispose … PLAYING` warning. The crash is resolved.

### Architectural implication

If clean teardown does not make this reliable, **running `pipewiresrc` inside
gnome-shell is too fragile** — and that is precisely why Hanabi and the
@kv9898 fork run GStreamer in a **separate process** and clone its window
instead of feeding the shell directly. Options, in order of preference:

1. Clean teardown makes it stable (this fix) — keep the in-shell consumer.
2. Receive PipeWire buffers in the shell with **libpipewire directly** (no
   GStreamer) and import dma-buf into Cogl — removes libgstpipewire entirely
   and converges with Stage C (Q-1).
3. Fall back to a separate GStreamer helper process (loses the no-extra-window
   differentiator; last resort).

Decide after the re-verification above. Until it is crash-free in both cases,
**no live-session test.**

## Fixes

See `extension/` changes and 40.03 (updated preconditions). Summary:

- **Never block startup.** All GStreamer work (init, device discovery,
  pipeline) is deferred off the enable path and runs after the shell is up;
  discovery is asynchronous and time-bounded; no producer → log and idle,
  retry slowly, never block, never throw.
- **`enable()` is fully guarded** — any exception leaves the shell untouched.
- **Kill-switch:** if `~/.config/wwb/disabled` exists, the extension no-ops at
  enable. Create it from a TTY to stop the grey-screen loop without removing
  the extension.
- **Churn cap (Failure 2 mitigation):** upload only when the actor is mapped,
  and document the per-frame allocation for the Stage-C zero-copy fix.
