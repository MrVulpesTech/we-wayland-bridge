# 40.03 — Live-session test runbook

The headless pipeline is proven (`40.02`). This runbook is for watching it on
the **real desktop**: Frieren (or any scene) animating as the wallpaper,
behind the icons. **The operator runs every step here.** The assistant must
never launch, enable, or kill anything against the live session (it shares
the operator's session — a stray `pkill gnome-shell` once caused a logout).

## ⚠️ Incident on the first attempt — read this first

The first live test caused a **hard freeze** and then a **grey-screen login
loop** (full write-up: `40.04`). Root cause: the extension ran GStreamer
init/discovery synchronously on the shell main thread at login **with no
producer running**, blocking the shell; because it was in `enabled-extensions`
it recurred every boot, so recovery needed physically removing the extension.

**Do not retry the live session until all of these hold:**

1. The hardened extension has been **re-verified in a nested session**
   (40.04 §repro): it must come up responsive **with no producer running**,
   logging `wwb: producer "wpe-host" not found yet` and idling — never grey.
2. You start the **producer manually first**, and only then enable the
   extension. Never rely on auto-enable-at-login as the path.
3. You know the **kill-switch**: `touch ~/.config/wwb/disabled` (from a TTY if
   needed) makes the extension no-op at the next enable/login. Use it to break
   any bad loop without removing files.
4. Keep the producer at the monitor's resolution, and expect heavier GPU load
   at native size (e.g. 2560×1440) — a suspected contributor to the freeze
   (40.04 Failure 2).

## Why this is reasonably safe

The extension is contained: it overrides one method
(`BackgroundManager._createBackgroundActor`) through `InjectionManager`, which
**reverts on disable**, and every frame path is wrapped in try/catch. A JS
exception makes GNOME disable the extension, not crash the shell. The
realistic worst case is a visual glitch, recoverable with one command
(§Recovery). Read §Recovery *before* starting so the path is muscle memory.

## Prerequisites

- Producer built: `bridge/build/host/wpe-host` (`bridge/build.sh`).
- Extension installed (symlink):
  ```sh
  ln -sfn "$PWD/extension" \
    ~/.local/share/gnome-shell/extensions/we-wayland-bridge@we-wayland-bridge.github.io
  ```
- A scene wallpaper id. Astronaut `2804205787` is 2560×1440 (16:9) and fills a
  1080p monitor cleanly; Frieren `3622495963` is 5160×2160 (ultrawide) and
  will be cropped on a 16:9 monitor (fit modes are a follow-up — `40.02` §5).
- Match the producer `--size` to the monitor resolution, and keep it
  **≥1080p** (low-res scenes wash out — Q-12).
- Everything from a normal terminal, **not** the snap-confined VS Code one.

## Start order

**1. Start screen recording first** (so the demo video isn't forgotten).
   Pick one:
   - GNOME built-in: open the screenshot UI (PrtSc), switch to the
     **record** mode, record the whole screen. Stop from the top-bar
     indicator.
   - `wf-recorder -f ~/wwb-demo.mp4` (Wayland-native; `Ctrl+C` to stop).
   - OBS, if installed.

**2. Start the producer.** Either:
   - Plain shell (simplest), leave the terminal open:
     ```sh
     cd ~/projects/we-wayland-bridge
     bridge/build/host/wpe-host --pipewire --assets-dir steam-assets \
       --size 1920x1080 --fps 60 steam-workshop/2804205787
     ```
   - Or a transient systemd scope (clean lifecycle; stop with
     `systemctl --user stop wpe-producer`):
     ```sh
     systemd-run --user --scope --unit=wpe-producer \
       ~/projects/we-wayland-bridge/bridge/build/host/wpe-host --pipewire \
       --assets-dir ~/projects/we-wayland-bridge/steam-assets \
       --size 1920x1080 --fps 60 \
       ~/projects/we-wayland-bridge/steam-workshop/2804205787
     ```
   It prints `PipeWire source running …` and `stream state: paused`. The node
   id does not matter — the extension finds it by `node.name=wpe-host`.

**3. Watch the shell log** in another terminal:
   ```sh
   journalctl --user -f -o cat | grep -i wwb
   ```
   (If that shows nothing, try `journalctl -f -o cat /usr/bin/gnome-shell | grep -i wwb`.)

**4. Enable the extension:**
   ```sh
   gnome-extensions enable we-wayland-bridge@we-wayland-bridge.github.io
   ```

## Expected

- The desktop background becomes the animated scene, behind the icons.
- The producer terminal flips to `stream state: streaming` and `producing …`.
- The log shows, once:
  ```text
  wwb: enabled (N monitors, N wallpaper actors attached)
  wwb: video source "Wallpaper Engine bridge" node.name=wpe-host
  wwb: CoglContext via …
  wwb: first frame uploaded (WxH)
  ```

## Recovery (read before starting)

**Level 1 — desktop responsive, looks wrong.** In any terminal:
```sh
gnome-extensions disable we-wayland-bridge@we-wayland-bridge.github.io
```
The override reverts and the normal background returns. Stop the producer
(`Ctrl+C`, or `systemctl --user stop wpe-producer`).

**Level 2 — shell misbehaving / grey screen / desktop unusable.** Switch to a
TTY: `Ctrl+Alt+F3` (try F4/F5), log in, then engage the **kill-switch** (the
simplest fix — the extension no-ops on the next login without touching files):
```sh
mkdir -p ~/.config/wwb && touch ~/.config/wwb/disabled
```
Belt-and-braces, also make it unloadable and disabled:
```sh
rm -f ~/.local/share/gnome-shell/extensions/we-wayland-bridge@we-wayland-bridge.github.io
gnome-extensions disable we-wayland-bridge@we-wayland-bridge.github.io 2>/dev/null || \
  dconf write /org/gnome/shell/enabled-extensions \
    "$(dconf read /org/gnome/shell/enabled-extensions | sed "s/'we-wayland-bridge@we-wayland-bridge.github.io', \?//")"
```
The kill-switch file alone is enough for the hardened extension; removing the
symlink guarantees the shell cannot load it at all. Switch back with
`Ctrl+Alt+F2` (or F1); if the session is grey, log out from the TTY
(`loginctl terminate-user $USER`) and log in fresh — with the kill-switch set,
it will not re-break.

**Level 3 — full reset.** Log out and back in (user menu → Log Out, or from a
TTY `loginctl terminate-user $USER`). With the symlink removed (Level 2), the
extension will not re-apply. On Wayland a shell crash ends the session
anyway; logging back in is a clean state.

Never `pkill gnome-shell` to "fix" it — on Wayland that ends the whole
session (it is what caused the earlier accidental logout).

## After a successful demo

- Stop the recording; keep the clip (this is the project's proof video).
- `gnome-extensions disable …` and stop the producer when done.
- Note any issues (tearing, fps, fullscreen behaviour, multi-monitor) for the
  follow-ups in `40.02` §5.
