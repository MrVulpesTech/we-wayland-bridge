> **Status: Session 2 findings.** Maps the renderer's output abstraction

## 1. The abstraction

The renderer separates "what runs the GL context and render loop" from
"where the frames go". Three cooperating types under
`src/WallpaperEngine/Render/Drivers/`:

| Type | File | Role |
|---|---|---|
| `VideoDriver` | `VideoDriver.h` | Owns the GL context, the window (if any), and the per-frame loop (`dispatchEventQueue`). Abstract. |
| `Output` | `Output/Output.h` | A render target: its viewports, size, and how a frame is presented (`updateRender`). Abstract. |
| `OutputViewport` | `Output/OutputViewport.h` | One screen/region: `makeCurrent()` (activate context) + `swapOutput()` (present). Abstract. |

Concrete implementations:

| Driver | Output(s) | Context | Present path |
|---|---|---|---|
| `GLFWOpenGLDriver` | `GLFWWindowOutput` (preview), `X11Output` (X11 root) | GLFW, hidden window (`GLFW_VISIBLE=GLFW_FALSE`), OpenGL 3.3 core | `glfwSwapBuffers`; X11 path additionally `glReadnPixels` → CPU buffer → X root |
| `WaylandOpenGLDriver` | `WaylandOutput` | EGL on `EGL_PLATFORM_WAYLAND_EXT`, `wl_egl_window` per output | `eglSwapBuffers` to a `wlr-layer-shell` surface |

## 2. How a backend is selected

Drivers **self-register** at load time via a constructor function:

```cpp
__attribute__ ((constructor)) void registerGLFWOpenGLDriver () {
    sVideoFactories.registerDriver(
        ApplicationContext::DESKTOP_BACKGROUND, "x11", /* factory lambda */);
    sVideoFactories.registerDriver(
        ApplicationContext::EXPLICIT_WINDOW, DEFAULT_WINDOW_NAME, /* ... */);
    // ...
}
```

`VideoFactories` keys factories on `(WINDOW_MODE, XDG_SESSION_TYPE)`:

- `WINDOW_MODE` ∈ { `NORMAL_WINDOW=0`, `DESKTOP_BACKGROUND=1`,
  `EXPLICIT_WINDOW=2` } (`ApplicationContext.h`).
- For window modes the session type is the literal `"default"`; for
  `DESKTOP_BACKGROUND` it is the real `XDG_SESSION_TYPE` (`"x11"` /
  `"wayland"`).

At startup the app asks `createVideoDriver(mode, sessionType, ...)`. On
GNOME Wayland the only `DESKTOP_BACKGROUND` + `"wayland"` driver is the
`wlr-layer-shell` one, which Mutter rejects — this is the failure GNOME
users hit (upstream #302, #452, #578).

**Implication:** a new backend is added by dropping in a new file whose
constructor registers a factory. No existing file is edited (principle
P2). The remaining question is which `(WINDOW_MODE, sessionType)` key it
claims, or whether it adds a new `WINDOW_MODE` plus a CLI flag.

## 3. The render loop already has a "read-back" path

`GLFWOpenGLDriver::dispatchEventQueue()` does, each frame:

1. `glClear`
2. update every viewport (`getApp().update(viewport)`)
3. **if `Output::haveImageBuffer()`** → `glReadnPixels(..., GL_BGRA, ...)`
   into `Output::getImageBuffer()` (a CPU-side buffer)
4. `Output::updateRender()` (the Output pushes that buffer wherever it
   goes — e.g. the X11 root pixmap)
5. `glfwSwapBuffers` + `glfwPollEvents` + FPS sleep

So a **copy-based** output already exists end to end: render off-screen
into a hidden GLFW window, read the pixels back to CPU, hand them to an
`Output`. This is the same data shape the memfd fallback in ADR-0004
describes — it is the cheapest path to a working frame, at the cost of a
GPU→CPU readback per frame.

## 4. Two ways to build the PipeWire backend

### Option A — copy path (low risk, fallback-grade)

Add a `PipeWireOutput : Output` that returns `haveImageBuffer() = true`,
allocates a buffer, and in `updateRender()` publishes that buffer as a
PipeWire stream with `memfd`/SHM. Reuses the GLFW driver and the existing
read-back loop wholesale. Gets a frame to GNOME quickly, but copies every
frame through the CPU (violates principle P4 as a permanent design; fine
as the documented degraded mode / first milestone).

### Option B — zero-copy path (the real goal, the risk)

Add a headless EGL `VideoDriver` that:

1. Creates a **surfaceless / off-screen** EGL context (e.g.
   `EGL_PLATFORM_SURFACELESS_MESA`, or gbm-backed), instead of the
   Wayland-window EGL the current driver uses (its config demands
   `EGL_SURFACE_TYPE = EGL_WINDOW_BIT` and a `wl_egl_window`).
2. Renders the scene into an **FBO** backed by a texture.
3. Exports that texture as a **dma-buf** (`EGL_KHR_image_base` +
   `eglExportDMABUFImageMESA`, or gbm bo export).
4. Publishes the dma-buf to **PipeWire**; the viewport's `swapOutput()`
   queues the new buffer instead of `eglSwapBuffers`.

This is the GPU-to-GPU path of ADR-0004 and the project's central
unproven risk (Q-1). The building blocks are present on the reference
machine: EGL is already a renderer dependency, and `libgbm-dev`,
`libpipewire-0.3-dev`, `libspa-0.2-dev` are installed.

Recommended sequence: build A to de-risk the GNOME extension and the
PipeWire plumbing with a guaranteed-correct frame, then build B and swap
the transport underneath, keeping A as the fallback.

## 5. Fullscreen detectors

`FullScreenDetector` subclasses are also factory-registered by session
type and drive the pause-on-fullscreen behaviour. For the GNOME path the
extension can own pause logic on the consumer side (principle P6/P7), so
the new backend likely needs no detector — to confirm when wiring the
extension (Session 3+).

## 6. Upstream is refactoring toward exactly this — PR #609

Checked upstream issues/PRs (Q-2). There is **no existing PipeWire or
dma-buf video-output backend** — every `pipewire` hit concerns *audio*.
But there is a development that changes the plan:

**PR #609 — "refactor: separate linux-wallpaperengine into smaller
components"** (branch `next-v2`, draft, ~379 files, updated 2026-06-11).
Its stated goals, verbatim from the PR:

- "Separate the render code from the window/desktop management."
- "Provide a stable library (`linux-wallpaperengine-core`) that other
  apps / desktop environments / etc can use to display any wallpaper
  engine background wherever they want."
- "Isolate desktop-environment code."
- A `dev-viewer` and the ability to "record multiple backgrounds at once"
  (implies an off-screen capture path already exists in the branch).
- "This is also part of the groundwork to start supporting DEs like GNOME
  or KDE in the short term."

This is the embeddable-core foundation this project's bridge needs,
being built by upstream right now. The structure mapped in §1–§3 above is
the **current `main`**; `next-v2` reorganizes it.

### Consequence for the project

- Building a backend against current `main`'s `Drivers/Output` risks
  being invalidated by the merge. The work should target **`next-v2`**
  and the `linux-wallpaperengine-core` library surface, not `main`.
- The integration may shift from "add an output driver inside the
  renderer" to "**consume `linux-wallpaperengine-core` from a small
  host program** that renders headless and publishes PipeWire" — which
  would shrink the renderer-side patch and strengthen the licence
  boundary (the core lib stays GPLv3; our host is a separate program).
- This reopens **ADR-0001** (submodule pinned to `main` → consider
  tracking `next-v2`) and **ADR-0003** (extend `Drivers/Output` → build
  on the core library). Both are flagged for revision; see Q-9.
