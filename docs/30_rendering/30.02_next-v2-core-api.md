## 1. Where the core library actually lives

The separate repository named in chat — `Almamu/linux-wallpaperengine-core`
— **does not exist publicly** (HTTP 404 / no API record as of
2026-06-11). The core library currently lives **in-tree** in `next-v2`
at `src/core/`, built by `add_subdirectory(src/core)` as the target
`linux-wallpaperengine-core`. It already generates an installable CMake
package config (`linux-wallpaperengine-core-config.cmake`) and a
pkg-config file (`linux-wallpaperengine-core.pc`), so it is designed to
be installed and consumed by external programs via `find_package` /
`pkg-config`. Extraction into its own repo looks planned but has not
happened yet (compare `Almamu/linux-wallpaperengine-window`, already a
separate repo).

The `next-v2` tree splits into:

```text
src/core/        the embeddable renderer library (GPLv3)
src/frontend/    the desktop app: links core, owns DE integration
src/dev-viewer/  a minimal host + an mp4 "recorder" (offscreen evidence)
src/webhelper/   web/CEF helper
include/linux-wallpaperengine/   the public C API headers
docs/support/    per-DE notes (GNOME, KDE, Sway, Hyprland, COSMIC, ...)
```

## 2. The public API is a C embedding interface

`include/linux-wallpaperengine/` exposes a C ABI (`extern "C"`, opaque
`void*` handles — `wp_context`, `wp_project`, `wp_configuration`). The
host program, not the library, owns the GL context and the output. Key
entry points:

| Call | Role |
|---|---|
| `wp_config_create` / `wp_config_set_*` | Build configuration (assets dir, backgrounds dir, audio, properties, pause/mute callbacks). |
| `wp_context_create(config)` | Lifecycle root. |
| `wp_context_set_gl_proc_address(ctx, cb)` | **Host supplies the GL function loader** — the host owns the GL/EGL context. |
| `wp_context_set_time_counter(ctx, cb)` | Host drives the clock. |
| `wp_context_set_audio_input_mix(ctx, cb)` | Host feeds PCM frames for audio-reactive scenes. |
| `wp_project_load_id / _folder` | Load a wallpaper. |
| `wp_project_hint_size(p, w, h)` | Host sets render resolution. |
| **`wp_project_set_output_framebuffer(p, fbo)`** | **Host hands core the GL framebuffer object to render into.** |
| `wp_render_update_time(ctx)` + `wp_render_frame(p)` | Render one frame into that FBO. |
| `wp_project_set_mouse_input` / `wp_project_notify_*` | Mouse + MPRIS-style now-playing metadata via host callbacks. |

This **answers Q-9's central question**: there is an explicit offscreen
render entry point. Core renders into a host-supplied FBO; the host owns
the texture backing it and decides what to do with the result.
Presentation is **not** internal to the library.

## 3. Confirmed in two working hosts

Not just headers — two callers in the branch drive exactly this loop:

`src/dev-viewer/.../SimpleBackgroundViewer.cpp`:

```text
wp_context_set_gl_proc_address(context, &glfw_gl_proc_address);
glGenFramebuffers(1, &framebuffer);
glBindFramebuffer(GL_FRAMEBUFFER, framebuffer);
glFramebufferTexture2D(GL_FRAMEBUFFER, GL_COLOR_ATTACHMENT0, GL_TEXTURE_2D, texture, 0);
wp_project_set_output_framebuffer(project, framebuffer);   // host-owned FBO
... per frame: wp_render_frame(project); glBindFramebuffer(0); // then present
```

The dev-viewer also has a `RecordingSession` (`recording.cpp`) that reads
the rendered frames and encodes them to mp4 with FFmpeg — i.e. an
offscreen render-to-texture pipeline with no visible-window dependency
already exists and is exercised.

`src/frontend/.../Desktop/Output.cpp` does the same (FBO + texture +
`wp_project_set_output_framebuffer` + `wp_project_hint_size` +
`wp_render_frame`), then presents the texture per desktop environment
(`Desktop/{Wayland,X11,Universal}/`). The Wayland presenter still uses
`wlr-layer-shell`, so the frontend itself still has no GNOME path
(`docs/support/GNOME.md` says only: "Doesn't support wlr-layer-shell").

## 4. Why this matters for us

Because core renders into a **host-allocated texture**, the host can back
that texture with a dma-buf (allocate via gbm/EGLImage, bind with
`glEGLImageTargetTexture2DOES`, attach to the FBO). Core renders straight
into the dma-buf; the same dma-buf is published to PipeWire. That is the
**zero-copy path of ADR-0004, with no renderer-internal changes** — the
producer becomes a separate host program linking core, not a patch to the
renderer's output drivers.

This largely **supersedes the `Drivers/Output` plan** from 30.01 (which
mapped current `main`). On `next-v2` the integration point is the C API,
not an internal driver class.

## 5. Maturity and risks (assessed honestly)

| Aspect | Finding |
|---|---|
| Branch status | **Draft PR**, not merged to `main`. Updated actively (2026-06-11). ~379 files changed. |
| Core repo | Not yet extracted to its own repo; lives in `next-v2/src/core`. |
| API shape | C ABI with opaque handles + callbacks — a deliberately stable surface, but still pre-merge and subject to change. |
| Build | Not built in this session (heavy: CEF, glad, FFmpeg). Builds in upstream CI per the PR; local build deferred to a later session. |
| Tests | Present but modest: parsing/library/playlists/mouse-coords (`tests/Tests/`). No rendering-path unit tests. |
| GNOME | No frontend GNOME path yet; the core embedding API is the intended foundation for one. |

**Risk:** building against `next-v2` means tracking a moving, unmerged
branch and a not-yet-extracted library. The API could shift before it
lands. Mitigation: depend on the narrow C surface (§2), keep our code on
the consumer side of a stable PipeWire contract (§6), and pin to a
specific `next-v2` commit until it merges.

## 6. Recommendation matrix for ADR-0001 / ADR-0003

This is input for revising those ADRs in a later session (no ADR rewrite
this session). Three layers:

### (a) Build now, throwaway — the PipeWire contract validator

The consumer side is independent of how frames are produced. Build it
first, against a **dummy producer** (a `videotestsrc` or a trivial
GStreamer PipeWire source), not the renderer:

- A GNOME Shell extension that consumes a PipeWire stream into a Clutter
  actor in the background layer (Session 3, as planned).
- This validates the hardest GNOME-specific unknowns (background-layer
  actor, dma-buf import into Cogl, fullscreen pause) with zero renderer
  coupling.

### (b) Wait on `next-v2` for — the producer

The frame **producer** should target the core embedding API, not a
renderer patch:

- A small host program: create a headless EGL context, allocate a
  dma-buf-backed texture + FBO, `wp_project_set_output_framebuffer`,
  `wp_render_frame`, publish the dma-buf to PipeWire.
- Hold this until we have either a pinned `next-v2` commit we trust or
  the merge lands. Until then, prototype the producer against the
  dev-viewer pattern to learn the API.

### (c) The stable seam — our PipeWire contract

Define and freeze our own **PipeWire stream contract** (pixel format, set
of dma-buf modifiers with a memfd/SHM fallback, resolution/refresh
handling, and any metadata such as pause state). Everything renderer-side
is then a **replaceable adapter** behind that contract:

```text
[ producer adapter ]  →  PipeWire contract  →  [ GNOME extension consumer ]
   options for the adapter:
   - main + glReadnPixels (copy, throwaway)
   - next-v2 core host + dma-buf (target)
   - upstream's own future official path
```

If upstream ships an official GNOME integration (see
`90_upstream/302-context.md`), our consumer and contract survive; only the
adapter is dropped. This keeps the project useful and non-duplicative
regardless of which producer wins.
